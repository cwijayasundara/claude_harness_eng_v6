# Graph navigation and resumable execution

The harness treats repository understanding as maintained state, not a repeated full-tree search. Its navigation loop connects the [control plane](../architecture/control-plane.md) to [product routes](product-routes.md): source writes mark the graph dirty, stop hooks refresh derived navigation, and brownfield routes retrieve a bounded context pack before reading broadly.

## The navigation feedback loop

```text
Write/Edit/MultiEdit
  -> verify-on-save records changed indexed paths in graph-dirty.jsonl
  -> Stop or SubagentStop invokes graph-refresh
  -> code-graph.json and derived navigation artifacts are refreshed
  -> context-pack/nav-query ranks symbols, graph neighbors, co-change, tests
  -> token-advisor checks future reads/searches against that context receipt
```

`graph-refresh.js` uses a lock and patches an AST-backed `specs/brownfield/code-graph.json` for dirty files. It then regenerates or updates symbol maps, deterministic DeepWiki material, graph indexes, concept/navigation artifacts, and the human-facing codebase page. If graph tooling is unavailable, it fails visibly rather than presenting an invented graph; a placeholder graph is upgraded on the first source write.

This loop **supplies the evidence** used by `/feature` and `/change` to choose seams, avoid dependencies, select tests, and maintain their committed brownfield wiki. It is distinct from the OpenWiki generated by this run, which documents this harness repository under `openwiki/`.

## Context-first change work

When a real brownfield graph exists, `/change` requires `context-pack` before large production reads or unconstrained repository searches:

```bash
node .claude/scripts/context-pack.js --diff --budget 1600 "<goal>"
```

The pack combines lexical/wiki ranking, optional semantic indexing, graph neighbors, co-change evidence, optional diff boost, impact-scoped test suggestions, and `read_next` symbol ranges. A route should read those ranges or skeletons first, rather than opening full large files. Low-confidence/no-match results require clarification or one narrow re-pack/search rather than an expanding exploration loop.

`nav-query.js` is the facade for pack, symbols, callers, impact, co-change, semantic retrieval, and refresh. The optional MCP server exposes a related navigation surface. These tools **constrain workflow routing** in [product routes](product-routes.md), especially brownfield seam selection and preservation work.

## Token governance

`token-advisor.js` consumes graph ranges and the latest context-pack receipt. It detects skipped context search, avoidable broad reads, and likely verbose commands. It is advisory by default but can block in enforced mode; this Project-Zero manifest sets `token_governor.mode` to `enforced`.

The intended hierarchy is:

1. use context pack and its cited ranges;
2. use symbol/caller/impact queries when a question remains;
3. read a narrow source slice;
4. fall back to broader inspection only when the graph is missing, stale, or insufficient.

The advisor must still fail open when no usable graph ranges exist. Token governance improves retrieval discipline; it does not replace source verification or safety checks in the [control plane](../architecture/control-plane.md).

## Resuming autonomous work

The harness has two recovery layers.

### In-session stop continuation

`auto-continue-on-stop.js` reads `claude-progress.txt` and `features.json`. When an eligible `/auto` run still has work, it asks the next turn to resume. It tracks passing-feature progress and stops issuing continuation nudges after a bounded no-progress watchdog, producing an explicit stuck condition instead of an infinite loop.

### Cross-process build chain

`build-chain.js` creates fresh `claude -p` processes around persisted state: plan-only build, repeated `/auto --once` build links, and finalization. It bounds links/no-progress attempts and checks budget state. A killed session can therefore resume from committed artifacts and the progress file rather than relying on the original context window.

This is why a route’s durable artifacts—requirements, stories, design, contracts, `features.json`, and progress logs—matter operationally. The [verification system](../quality/verification.md) independently checks many of them for traceability and coverage.

## Change guidance

- Keep graph output deterministic: tests cover dirty marking, incremental patches, render behavior, stale stamps, and placeholder upgrades.
- Treat `code-graph.json` as an input to multiple consumers: navigation, coupling/cycle sensors, ownership/impact analysis, brownfield docs, and context guidance. A format or refresh change needs cross-consumer review.
- Preserve graph-refresh locking and no-hang behavior; stop hooks run in both session and subagent termination paths.
- When modifying context ranking or token enforcement, test both genuine governance and false-positive behavior; broad-read blocking must not make unknown/no-graph repositories unusable.
- For lane behavior and explicit context requirements, follow links back to [product routes](product-routes.md).
