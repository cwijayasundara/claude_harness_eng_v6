Claude Harness Engine · Overview
Claude Code plugin GAN + Ratchet 2.1.0

Build and change software with a trustworthy agent harness

The Claude Harness Engine is a Claude Code plugin that turns a raw coding model into a controlled engineering system: generator/evaluator separation, monotonic quality ratchets, session chaining, brownfield maps, and human-owned merge.

Mental model

Agent = Model + Harness. The model writes code; the harness makes the output trustworthy.

Guides steer before the agent acts; sensors detect after; humans improve both over time (HARNESS.md).

Maintainability

Lint, types, length caps, coverage ratchet, modularity review, mutation smoke.

Architecture

Layer imports, cycles, unstable hubs, API contract drift, topology presets.

Behaviour

Unit tests, API/Playwright/vision eval, security, ATs, regression suites.

Traceability

FRD→BRD→spec→test grounding, ownership, canvas sync, verification matrix.

Core law

The generator never grades itself. The evaluator never writes production code. Machine gates always stay on — even in full-auto mode. Humans always own merge.

Pick your lane

The harness should classify work before it edits. When unsure, describe the request plainly and let routing decide.

What are you doing? │ ├─ Installing harness into a repo ──────────────► /scaffold │ ├─ Building something NEW │ ├─ Small (CLI, lib, ≤5 stories) ─────────────► /build --lite │ ├─ Normal PRD, human gates ─────────────────► /build <prd> │ └─ Hands-off PRD→PR ────────────────────────► /build <prd> --auto │ ├─ Existing product, next full PRD ─────────────► /sprint <prd-file> │ └─ Existing codebase change ├─ Discovery only ──────────────────────────► /brownfield [--seams] ├─ Normal request ──────────────────────────► /feature "<request>" │ (routes to /vibe | /change | /refactor | epic path) ├─ Tiny safe edit (≤3 files, <150 LOC) ─────► /vibe ├─ Behavior change / bug ───────────────────► /change [--issue N] └─ Structure only ──────────────────────────► /refactor Always before merge ────────────────────────────► /gate
Public surface is intentionally small: /build · /feature · /sprint · /gate. Other commands are still available; the harness usually routes to them.
Mis-routing is the failure mode

Using /vibe for auth, migrations, or multi-story work burns time and skips the gates that would have caught the risk. Score the task on the Z/L continuum when the lane is unclear.

Behavior-preserving. Refactor commits stay refactor-free of behavior; purity gate enforces it.

Legacy preservation disciplines (auto-applied)

  • checking-coverage-before-change — know which tests cover a symbol before editing
  • pinning-down-behavior — characterization tests at the nearest seam for UNCOVERED symbols
  • sprouting-instead-of-editing — add new tested unit when god-files have no clean boundary
  • keeping-refactors-pure — separate structure from behavior commits
  • checking-migration-safety — expand/contract + reversibility for schema changes
  • writing-acceptance-tests-first — business-readable ATs against a Ports-and-Adapters seam

Gate & PR loop

/gate is the adaptive pre-merge quality gate (renamed from harness /review to avoid clashing with native PR review).

  • Deterministic checks (lint/types/coverage/architecture/regression…)
  • Evaluator (running app vs contract)
  • Fresh-context diff review (code-reviewer)
  • Security review when the diff crosses security/data/API boundaries — 3-instance majority vote
  • Always ends with quality-card + logical walkthrough + docs/CODEBASE.md refresh

/pr-respond <pr#> polls CI + review comments, classifies, fixes, pushes, replies with evidence — bounded and budget-metered, never merges.

System map

One scaffold, two runtimes, one work contract.

Neither runtime changes how /auto runs inside the workspace — only who picks the next group and where proof appears.

GAN architecture

Structural separation eliminates self-evaluation bias — the dominant failure mode of autonomous coding.

Biases removed by structure: confirmation bias, rationalization, scope creep. Max 3 self-heal attempts, then BLOCKED + learn.

Why this exists

Failure 1

Self-evaluation bias

Same agent writes and grades → rationalizes failures and skips edge cases.

Failure 2

Quality regression

Without ratchets, coverage drops and architecture drifts across iterations.

Failure 3

Context exhaustion

Projects exceed one window; without recovery, work restarts from scratch.

/auto loop — under the covers

The autonomous build loop: recover → negotiate → execute → ratchet → learn → chain.

Recovery ~700–1000 tokens/iteration reads: program.md · learned-rules.md · claude-progress.txt · features.json · dependency-graph.md

Stopping criteria

  • All features pass
  • 50 total iterations hard stop
  • 3 consecutive failed self-heals → BLOCKED, learn, move on
  • Budget caps (wall-clock / spawns / est. cost)

State files that matter

  • claude-progress.txt — append-only resume log
  • features.json — pass/fail registry
  • learned-rules.md — monotonic institutional memory
  • .claude/program.md — human steering input

Sprint contracts

Machine-readable definition of “done,” negotiated in exactly two calls: generator proposes → evaluator finalizes. Then immutable.

{
  "group": "C",
  "stories": ["E3-S1", "E3-S2"],
  "contract": {
    "api_checks": [{ "method": "POST", "path": "/documents/upload", "expected_status": 201 }],
    "playwright_checks": [{ "url": "/upload", "steps": [/* navigate, fill, click, assert */] }],
    "design_checks": { "visual_hierarchy": { "min_score": 7 } },
    "performance_checks": [{ "endpoint": "/documents/upload", "max_response_time_ms": 2000 }]
  }
}
Integration tests must round-trip real artifacts

Never hand-build fixtures that skip the real validator. A fixture with the wrong shape keeps tests green while the feature is inert. Use the real contract schema + validate-contract.

Three-layer evaluation (+ security)

The evaluator never reads source code. It only runs the application and checks observable behavior.

Verification modes: docker (default) · local · stub (mock from api-contracts schema). On fail, Docker logs / stderr feed the generator exact stacks.

Design-critic score: (DQ×1.5 + O×1.5 + C×0.75 + F×0.75) / 4.5. Plateau across 3 iterations forces a pivot.

AT-first layer sits below E2E: fast business-readable acceptance tests through a Ports-and-Adapters seam with test-double adapters — primary per-story loop; Playwright remains the full-stack confirmer.

Fail open is never silent

If a toolchain is missing or times out, the gate prints WARNING: GATE SKIPPED — so a silent skip is never mistaken for a pass. Hook crashes append to .claude/state/hook-errors.log.

Receipt-proof disciplines

Prompt-only discipline fails under pressure. These gates require on-disk receipts — same pattern for legacy edits and AT-first stories.

Legacy: check coverage → pin or sprout → edit

Sprouting is for god-files / low seam scores — new fully-tested unit instead of editing legacy in place. Pure refactors use keeping-refactors-pure + refactor-purity gate.

AT-first: red receipt before new production code

An AT that passes against unimplemented code proves nothing — record-at-red refuses to write a receipt on green.
Receipt files (usually gitignored under reviews)

specs/reviews/coverage-verdicts.jsonl
specs/reviews/at-red-receipts.jsonl
Session/CI can still prove presence at commit time from the working tree.

Known limitation (AT-first)

Receipt proves the AT was red by commit time — not strict chronological order vs every later implementation hunk. Still far stronger than prompt-only discipline.

Harness vs Devin — multi-dimension comparison

Different products, overlapping jobs. Devin (Cognition) is a hosted autonomous software engineer with its own cloud workspace, IDE, browser, DeepWiki, and team UI. This harness is an open control system for Claude Code: guides, sensors, GAN roles, ratchets, and explicit human merge — running in your repo and CI.

How to read this

Not a scoreboard to “win.” Devin optimizes for delegating engineering work to a cloud agent product. The harness optimizes for making Claude Code runs trustworthy, resumable, and governed inside an existing SDLC. Teams often use complementary patterns: product agents for parallel backlog crush; harnesses for monorepo discipline and merge-ready proof. Devin product details below track public docs (e.g. docs.devin.ai); both products evolve quickly.

Devin

AI software engineer product

Cloud sessions · embedded IDE · shell · browser · Slack/Teams · DeepWiki · playbooks · API/CLI · parallel backlog work.

This harness

Control plane on Claude Code

Open plugin · generator≠evaluator · sprint contracts · pre-commit registry · living code-map · /feature|/build|/gate · human-owned merge.

Shared DNA

Long-running agent work

Both care about repo understanding, multi-step tasks, PRs, self-correction from failures, and not losing state across long sessions.

10. Strengths map (honest)

Harness tends to win when…

  • You need auditable, repo-owned gates every commit
  • Self-eval bias is unacceptable (GAN + binary contracts)
  • You want model tiering and open prompt surfaces
  • Brownfield needs pin/sprout, AT-first, amendment provenance
  • Air-gapped / strict data-boundary: code stays on your runners
  • You’re building a long-lived product with sprint evolution

Devin tends to win when…

  • You want a turnkey cloud engineer with IDE + browser UX
  • High parallel backlog intake (Slack/Jira → many sessions)
  • Migrations / bulk refactors as fire-and-review PRs
  • Org-wide Knowledge + DeepWiki without building a harness
  • Non-Claude-Code users need a full product UI
  • You value product velocity over owning the control plane
Neither replaces senior engineering judgment on Z-band work (auth, money, migrations, public API).

Ideas this harness shares with the Devin-era playbook

Without claiming feature parity or shared implementation:

  • Living wiki / search — DeepWiki-inspired navigation; harness keeps a deterministic graph + wiki in-repo
  • Long-running sessions — externalized progress (progress log, features.json) similar in spirit to durable agent sessions
  • PR as handoff — agents draft; humans merge
  • Knowledge that triggers — learned-rules / program.md vs Devin knowledge triggers & playbooks
  • Ask the codebase — cited answers before thrashing files

Compatible stack

Use Devin (or any agent) to produce a PR, then require harness /gate (or the same CI sensors) before merge. The harness’s value is the control system; Devin’s value is the agent product & workspace. They compose.

Z/L continuum — place each task

You are not “Team vibe” or “Team read every line.” Each task gets a band, then a lane and a human read plan.

L · 0–2/vibe or disposable · skim walkthrough
M · 3–5/change · /feature · contracts + hunks
Z · 6–10full gate · line-read risk surface

Score axes 0–2: blast radius · observability of done · longevity · coverage of touched symbols · ownership. Hard overrides to Z: auth, money, PII, migrations, public API, UNCOVERED god-files.

Playbooks — use this effectively

Concrete recipes for the patterns that show up most.

1. First product from a PRD

  1. Scaffold the empty or starter repo with /scaffold (topology: web-app or api-service).
  2. Write a good PRD (see docs/prd-format.md) — acceptance criteria must be testable.
  3. Run gated build /build docs/prd.md and approve BRD → stories → design.
  4. Let /auto run; watch /status; if BUDGET stops, resume.
  5. /gate then read walkthrough + quality-card before merge.

2. Brownfield feature on a living system

  1. Ensure maps are fresh (or let /feature refresh them).
  2. /feature "…" — do not jump straight to editing.
  3. Context-pack before deep reads; pin or sprout UNCOVERED symbols.
  4. Ship PR with gate receipts; use /pr-respond for CI/comments.

3. Risky seam (auth, migration, god-file)

  1. /brownfield --seams "…" to rank cut-points.
  2. Score Z/L — expect Z; plan whole-branch review.
  3. Prefer sprout or expand-contract over editing uncovered cores in place.
  4. Full /gate with security path; majority re-verify when boundary fires.

4. Sprint N evolution

  1. Author next PRD against what already shipped.
  2. /sprint docs/prd-sN.md — approve requirement delta + design amendment only.
  3. Never regenerate design wholesale; amendments keep provenance.

5. Cheap day-to-day hygiene

  • Steer via .claude/program.md between sessions
  • Apply suggested CLAUDE.md learnings between sessions only
  • npm run drift on a schedule; copy harness-drift workflow if needed
  • /agent-readiness monthly; raise baseline when pillars improve
  • Upgrade control plane with /scaffold-upgrade --apply without wiping state
